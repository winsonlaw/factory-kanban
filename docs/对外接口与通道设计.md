# 对外接口与通道设计

> 版本 v1.0 · 配套契约：`packages/data-platform/src/contracts/`
> 本文定义系统的**两个对外边界**：南向（设备接入）与 MES 对接。
> 内部的 WebSocket 推送（data-platform → 大屏）见《产品规划方案》，本文不展开。

---

## 一、接口全景

```
                         ┌─────────────────────────────────────────┐
   南向 / 设备侧          │            data-platform (中心)            │      MES 对接
                         │                                           │
⚙️ 设备 ──Modbus/OPC-UA  │  ┌─────────┐   ┌──────────┐   ┌────────┐ │  🔗 MES
  /MQTT/SECS/RS485──▶ 边缘│  │ MQTT 接入│─▶│ 计算引擎  │─▶│MES适配器│◀┼──REST 拉工单
  网关 ──canonical/MQTT──▶│  │ (EMQX)  │   │OEE/Pacing│   │REST/WH │─┼──Webhook 回传
                         │  └─────────┘   └──────────┘   └────────┘ │
                         └─────────────────────────────────────────┘
   通道①：边缘→中心 MQTT/TLS              通道②：中心↔MES REST+Webhook
```

| 边界 | 方向 | 通道 | 协议 | 契约 |
|------|------|------|------|------|
| **① 设备接入** | 设备 → 边缘 | 现场总线 | Modbus/OPC-UA/MQTT/SECS-GEM/RS485 | 各设备原始 |
| **① 设备接入** | 边缘 → 中心 | MQTT over TLS | EMQX | `DeviceTelemetry` / `DeviceEvent` |
| **② MES 对接** | MES → 中心 | REST 拉取 + Webhook 推送 | HTTPS/JSON | `MesWorkOrder` |
| **② MES 对接** | 中心 → MES | Webhook 回调 | HTTPS/JSON | `MesEventEnvelope` |

## 二、通道① 南向：设备接入

### 2.1 两段式接入

```
设备 ──①现场协议──▶ 边缘网关 ──②canonical/MQTT──▶ 中心
       差异在这吸收            统一在这之后
```

**关键设计：协议差异、设备差异全部在边缘网关「设备 Profile」消化，中心只认 canonical。**

### 2.2 现场协议适配（设备 → 边缘网关）

本平台不止工业设备——同一套 canonical 管道也接**家电 / 空调 / 传感 / 智控**等 IoT 设备（工业+IoT 并存）。

| 协议 | 典型设备 | 采集方式 | 状态 |
|------|---------|---------|------|
| **Modbus TCP/RTU** | 多数 PLC、温控、传感 | 轮询寄存器 | ✅ 驱动骨架 |
| **TCP 通用** | 自定义 TCP 上报设备 | socket + 帧解析(分隔符/定长/JSON) | ✅ 已实现 |
| **OPC-UA** | 西门子、B&R、贝加莱 | 订阅节点 | ⏳ |
| **MQTT** | 已联网智能设备 | 订阅设备主题 | ✅（中心侧） |
| **HiLink（华为）** | 家电、空调、智能家居 | productId+SN，云/网关桥接 | ✅ 模型+仿真 |
| **HTTP/REST** | 网关/设备主动上报或轮询 | 周期拉取 / 回调 | ✅ 配置就绪 |
| **CoAP / 轻量物联** | 电池类传感、模组 | observe / 轮询 | ✅ 配置就绪 |
| **SECS/GEM (HSMS)** | 半导体级贴片/AOI | GEM 状态/数据采集 | ⏳ |
| **RS485 串口** | 老旧设备 | 自定义帧解析 | ⏳ |

> 设备状态统一进**通用设备状态库**（`GET /api/devices`），工业设备另进 OEE 聚合。

### 2.3 设备 Profile（兼容不同设备类型的核心）

每类设备一个 Profile，声明「原始地址 → canonical 字段」的映射。新增设备型号 = 加一份 Profile，**不改代码**。

```yaml
# 示例：回流焊 Profile（Modbus）
deviceType: reflow_oven
protocol: modbus-tcp
poll: 1000ms
map:
  equipmentStatus: { register: 40001, transform: statusCode }
  passCount:       { register: 40010, mode: increment }
  cycleTimeMs:     { register: 40012 }
  metrics.zoneTempsC:    { register: 40020, count: 8, scale: 0.1 }
  metrics.conveyorSpeed: { register: 40030 }
  alarm.code:      { register: 40040, dict: reflow_alarm }
```

Profile 输出统一的 `DeviceTelemetry`，`deviceType` 决定 `metrics` 形状（见 `contracts/device.ts` 可辨识联合）。

### 2.4 边缘 → 中心：MQTT 主题与负载

> 契约：`contracts/mqtt-topics.ts`

```
上行  fk/v1/telemetry/{factory}/{zone}/{line}/{station}   遥测   → DeviceTelemetry
上行  fk/v1/event/{factory}/{zone}/{line}/{station}        事件   → DeviceEvent
上行  fk/v1/status/{factory}/{zone}/{gateway}              心跳   → GatewayHeartbeat
下行  fk/v1/cmd/{factory}/{zone}/{line}                    指令   → DownlinkCommand
```

**遥测负载示例**（回流焊一次过机）：

```json
{
  "schemaVersion": "1.0",
  "messageId": "01J8X9...ULID",
  "timestamp": 1717300000000,
  "factoryId": "F01", "zoneId": "Z03", "lineId": "L02",
  "stationId": "S04", "deviceId": "DEV-RFL-001",
  "deviceType": "reflow_oven",
  "passCount": 1, "failCount": 0, "cycleTimeMs": 18000,
  "equipmentStatus": 1,
  "metrics": { "zoneTempsC": [150,180,210,245,250,230], "peakTempC": 250, "conveyorSpeedMmMin": 900 }
}
```

### 2.5 QoS、离线补传、去重

| 关注点 | 方案 |
|--------|------|
| **可靠投递** | MQTT QoS 1（至少一次）+ 中心按 `messageId` 幂等去重 |
| **断网补传** | 网关本地缓冲 30 分钟（SQLite），恢复后按时间序补发，`timestamp` 为业务时刻不被覆盖 |
| **乱序** | 中心以 `timestamp` 排序入库；遥测累加用增量值避免覆盖 |
| **安全** | MQTT over TLS + 每网关独立证书/账号；EMQX ACL 限制网关只能发自己车间主题 |
| **背压** | EMQX 规则引擎分流；中心批量写 TDengine |

### 2.6 下行指令（中心 → 边缘）

`reload_config`（重载 Profile/阈值）、`set_threshold`、`device_control`（透传设备控制，需权限）、`request_resync`（请求重传缓冲）。

## 三、通道② MES 对接（REST + Webhook）

> 契约：`contracts/mes.ts`。统一适配器接口 `MesAdapter`，REST 为默认实现。

### 3.1 双向数据

```
入站  MES ──工单/数量/交货时间──▶ 看板     （REST 拉取 + Webhook 通知）
出站  看板 ──生产进度/完工/质量/停机/换型──▶ MES   （Webhook 回调）
```

### 3.2 入站：获取工单（MES → 看板）

两种方式互补：

**A. 数据平台主动拉取（REST，增量轮询）**

```
GET /mes/orders?factoryId=F01&status=released,in_progress&updatedSince={ts}&page=1&pageSize=100
→ 200  MesOrderPage  { total, page, pageSize, items: MesWorkOrder[] }
```

- 频率：1–5 分钟增量轮询（`updatedSince` 只取变更）
- 适合：MES 不主动推、或作为 Webhook 的兜底对账

**B. MES 主动推送（Webhook，实时）**

```
POST {我方回调}/mes/webhook/orders     ← MES 调用
Body: MesOrderChangeNotice { noticeId, changeType, order }
→ 200  { accepted: true, noticeId }
```

- `changeType`: created / updated / rescheduled / cancelled
- 适合：排产/插单实时同步；`noticeId` 幂等去重

> **落地最小集**：先打通 A（拉取），再加 B（推送）做实时增强。

### 3.3 出站：回传生产消息（看板 → MES）

统一信封 `MesEventEnvelope`，MES 侧**单端点**接收，按 `eventType` 分发：

```
POST {MES 回调 URL}/production/events     ← 我方调用
Body: MesEventEnvelope
→ 200  MesAck { accepted, eventId, mesRef? }
```

| eventType | 触发 | data 契约 | 频率 |
|-----------|------|----------|------|
| `production.progress` | 周期上报产量 | ProductionProgressData | 每 1 分钟 / 每 N 件 |
| `production.completed` | 工单完工 | ProductionCompletedData | 完工即时 |
| `quality.report` | 质量/不良 | QualityReportData | 周期 / 阈值触发 |
| `equipment.downtime` | 停机 | DowntimeData | 事件 |
| `equipment.alarm` | 报警 | AlarmData | 事件 |
| `changeover` | 换型起止 | ChangeoverData | 事件 |

**进度上报示例**：

```json
{
  "schemaVersion": "1.0",
  "eventId": "01J8XB...ULID",
  "eventType": "production.progress",
  "occurredTs": 1717300060000,
  "source": { "factoryId": "F01", "zoneId": "Z03", "lineId": "L02" },
  "orderNo": "WO-20260602-001",
  "data": {
    "goodQty": 12, "failQty": 1,
    "cumulativeGoodQty": 998, "cumulativeFailQty": 28,
    "oee": 0.721, "uph": 720,
    "intervalStartTs": 1717300000000, "intervalEndTs": 1717300060000
  }
}
```

### 3.4 鉴权

| 方向 | 方案 |
|------|------|
| 我方调 MES | MES 颁发 API Key / OAuth2 Client Credentials；HTTPS |
| MES 调我方 Webhook | 共享密钥 **HMAC-SHA256 签名**（`X-Signature` 头）+ IP 白名单 + 时间戳防重放 |

### 3.5 可靠性：重试、幂等、对账

| 关注点 | 方案 |
|--------|------|
| **幂等** | 每条事件 `eventId` 全局唯一，MES 按此去重；我方重发不重复入账 |
| **重试** | 推送失败指数退避（1s→2s→4s…，上限 5 次），仍失败入死信队列 |
| **顺序** | 同工单事件按 `occurredTs` 排队串行推送，避免进度回退 |
| **对账** | 每班结束批量 `pushBatch` 补发当班全部事件 + 提供 `GET /mes/reconcile?shift=...` 比对总量 |
| **降级** | MES 不可用时事件落 PostgreSQL 待发队列，恢复后顺序补发，零丢失 |

### 3.6 适配器抽象（REST 为主，保留扩展）

上层业务只依赖 `MesAdapter` 接口，底层可换实现，落地时按客户 MES 选配：

```ts
interface MesAdapter {
  fetchOrders(query): Promise<MesOrderPage>   // 入站拉取
  pushEvent(event): Promise<MesAck>           // 出站单发
  pushBatch(events): Promise<MesAck[]>         // 出站批量/对账
}
```

| 实现 | 适用 | 状态 |
|------|------|------|
| `RestMesAdapter` | MES 提供 REST API（多数现代 MES） | **默认，本期** |
| `MqMesAdapter` | MES 走 Kafka/RabbitMQ | 预留 |
| `DbMesAdapter` | 老 MES，共享库中间表 | 预留 |

## 四、错误码与版本

### 4.1 统一错误体

```json
{ "error": { "code": "MES_ORDER_NOT_FOUND", "message": "...", "traceId": "..." } }
```

| code | 含义 |
|------|------|
| `AUTH_FAILED` | 鉴权失败 |
| `SIGNATURE_INVALID` | Webhook 签名错误 |
| `SCHEMA_MISMATCH` | 负载不符契约（含版本不兼容） |
| `DUPLICATE_EVENT` | 幂等命中（非错误，告知已处理） |
| `MES_UNAVAILABLE` | MES 暂不可用，已入待发队列 |

### 4.2 版本管理

- MQTT 主题含 `v1`、消息含 `schemaVersion`；REST 路径含 `/v1`（或 `Accept-Version` 头）。
- **加字段不升大版本**（消费端忽略未知字段）；**改语义/删字段才升**。
- 新旧版本并行灰度，按 `schemaVersion` 路由。

## 五、落地优先级

| 优先级 | 内容 |
|--------|------|
| **P0** | 边缘网关 Modbus Profile + canonical 上报 + EMQX 接入 + 幂等去重 |
| **P1** | MES 入站拉取（`RestMesAdapter.fetchOrders`）→ 驱动 Pacing/达成率 |
| **P2** | MES 出站 `production.progress` / `production.completed` 回传 |
| **P3** | 设备事件 + 停机/报警/换型回传 + 离线补传 + 对账 |
| **P4** | MES Webhook 入站（插单实时）+ 下行指令 + 多协议（OPC-UA/SECS） |
