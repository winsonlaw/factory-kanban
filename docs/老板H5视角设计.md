# 老板 / 管理层 H5 视角设计

> 版本 v1.0 · 形态:移动端 H5(竖屏)· 技术:React + antd-mobile · 范围:单厂老板(集团多厂为后续扩展)
> 这是继 主任/调度/组长/参观 之后的**第 5 个角色视角**,高度最高——经营视角。

---

## 一、定位与场景

| 维度 | 说明 |
|------|------|
| **使用者** | 工厂老板 / 总经理 |
| **设备** | 手机(微信/浏览器打开 H5) |
| **场景** | 碎片时间,一天扫几次;不盯屏、不救火(救火是调度屏) |
| **核心关切** | **今天能不能交货?赚不赚钱?哪条线拖后腿?要不要追责/调资源?** |
| **刷新** | 下拉刷新 + 30 秒自动轮询(非秒级实时) |

> 与车间主任屏的区别:主任屏在办公室大屏、盯当班达成;老板 H5 在手机、看**经营结果 + 趋势 + 钱**,弱化 OEE 细节、强化金额与交付。

---

## 二、设计原则

1. **从"OEE 数字"升级到"经营判断"**:每个数字给红黄绿 + 钱 + 落后多少。
2. **信息三层金字塔(竖屏)**:
   - **一眼层(首屏不滚)**:今日达成大灯 + 产值 + 欠产/落后
   - **三秒层(滚一屏)**:红黑榜、损失金额、订单交付、OEE 趋势
   - **深看层(点进去)**:单线详情、订单详情、异常清单
3. **金额化**:件数→钱(产值、停机损失、质量损失),老板对钱最敏感。
4. **移动优先**:大字、卡片、竖向滚动、拇指可达、下拉刷新。
5. **颜色语义统一**(与全系统一致):🟢达标/超前 🟡预警 🔴未达/落后 ⚪停机。

## 四、核心 KPI 词典

| # | 指标 | 公式 / 定义 | 数据来源 | 红黄绿 |
|---|------|------------|---------|--------|
| ② | **今日产值** | Σ各线产量 × 单件产值 | workshop.lines[].passCount × costPerUnit | — |
| ② | **今日达成率** | 实产 / 计划 | ΣpassCount / ΣtargetCount | ≥100绿/≥85黄/<85红 |
| ② | **欠产·落后** | 实产−应产;落后=欠产×Takt | 由 shiftStartTs+taktSec 算 Pacing | <0 红 |
| ③ | **月度达成率** | monthActual / monthTarget | workshop.monthActual/monthTarget | — |
| ③ | **同比** | 本月 vs 去年同期 | workshop.monthYoY | >0 绿 |
| ④ | **产线红黑榜** | 各线达成率排序 | lines[] 排序 | 自动红黑 |
| ⑤ | **停机损失¥** | 停机损失产能 × 单件产值 | exec/summary(新) | 金额越高越红 |
| ⑤ | **质量损失¥** | 不良数 × 单件成本 | exec/summary(新) | — |
| ⑥ | **订单交付** | 已完成/订单量 + 交付预测 | workshop.orders[] | risk=true 红 |
| ⑦ | **综合OEE趋势** | 近7天 + 目标线 | workshop.oeeTrend7d[] | <0.85 黄 |
| ⑧ | **待关注异常** | 停机久/连续不良/缺料 | workshop.alarms/materialAlerts | 按级排序 |
| ⑨ | **能耗** | Σ IoT 设备实时功率/累计电量 | /api/devices(smart_plug/ac powerW) | 同比异常预警 |
| ⑩ | **安全运行天数** | 距上次事故 | workshop.safeDays | — |

## 五、数据来源与接口

### 5.1 大部分复用现有接口

`GET /api/workshop/W01`(已实现)已包含:`lines[]`(产量/目标/OEE/状态/currentOrder)、`alarms[]`、`materialAlerts[]`、`changeovers[]`、`costPerUnit`、`monthTarget`、`monthActual`、`oeeTrend7d[]`、`orders[]`、`safeDays`、`monthYoY`。

→ **首屏达成、月度、红黑榜、订单交付、OEE趋势、异常、安全天数 全部可直接算。**

### 5.2 需新增一个经营聚合接口

```
GET /api/exec/summary?factoryId=F01
→ {
    todayOutputValue, targetValue,        // 产值金额
    downtimeLossAmount, qualityLossAmount, // 金额化损失
    downtimeTopReasons: [{reason, hours, amount}],
    laborEfficiency?, energyKwhToday?      // 人效/能耗(可选,接 IoT/考勤)
  }
```
- v1 可由 workshop 数据 + costPerUnit 在服务端计算(停机损失需补「停机时长」统计,见 reporter 的 downtime 事件)。
- **能耗**直接复用已建的 IoT 设备层:聚合 `/api/devices?deviceType=smart_plug` 的 `powerW/energyKwh`——把之前做的 IoT 接入用起来。

### 5.3 刷新方式

H5 非实时救火,推荐 **下拉刷新 + 30s 轮询 REST**(比 WS 省电、简单);需要更实时可选接 `WS /ws/workshop/W01`。

## 六、技术方案

| 项 | 选型 |
|----|------|
| 新包 | `packages/boss-h5` |
| 框架 | React 18 + Vite + TypeScript |
| 移动端 UI | **antd-mobile**(卡片/PullToRefresh/Steps/Progress) |
| 图表 | echarts-for-react(折线+目标线、进度环、迷你火花图) |
| 适配 | viewport meta + vw/rem,竖屏 375~430px 基准 |
| API | 复用 admin-web 的 fetch 客户端模式;老板视图只读 |
| 部署 | nginx 托管(同 admin-web 模式),微信/浏览器打开;加入 docker compose |
| 主题 | 浅色商务为主,可选深色 |

**权限**:建议新增只读角色 `boss` 或复用 `viewer`;经营接口 `/api/exec/*` 走 viewer+ 鉴权,或生成**分享链接 token**(老板免登录扫码看)。

## 七、交互

- **下拉刷新** + 30s 自动轮询;顶栏显示最后更新时间
- **卡片点击下钻**:产线→单线详情、订单→订单详情、异常→异常清单
- **工厂切换**:单厂隐藏,多厂时顶部 Tab(为 P3 预留)
- **分享**:H5 在微信内打开,支持截图/转发;关键卡可生成分享图
- **异常角标**:待关注数量红点

## 八、落地路线图

| 阶段 | 内容 | 依赖 |
|------|------|------|
| **P0** | 首屏:达成大灯 + 产值 + 月度环 + 红黑榜 | 仅 `/api/workshop` |

> P0 大部分用现成数据,**可最快出一个能在手机上打开的老板首屏**。

## 九、与现有体系的关系

```
                  ┌─ 大屏(TV):主任/调度/组长/参观 ── WS 实时
data-platform ────┤
   (一套数据)     └─ H5(手机):老板/管理层 ──────── REST 下拉刷新
                       复用 /api/workshop + /api/exec + /api/devices(能耗)
```

一套后端、一份数据,多种终端形态。老板 H5 是"经营层"的轻量出口。
