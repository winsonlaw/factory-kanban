# 管理后台设计（admin-web）

> 版本 v1.0 · 前端 `packages/admin-web`（React + Ant Design）· 后端配置域 `packages/data-platform/src/config-domain`
> 参考 ThingsBoard：资产层级 + Gateway 连接器（采集服务/通讯/数据点）。

---

## 一、核心实体关系

```
工厂 Factory
 └─ 车间 Workshop ──────────────────────────────── 组织域（资产树）
     └─ 产线 Line
         └─ 站位 Station  (绑定设备类型 DeviceProfile)
             └─ 采集服务 Collector ─────────────── 采集域（运行于某网关，绑定到站）
                 ├─ 通讯块 CommChannel  ← 协议+连接 (Modbus host:port / OPC-UA endpoint / MQTT)
                 └─ 数据块 DataPoint[]   ← 点表：寄存器/节点 → canonical 字段映射
```

**自底向上的挂接链**（用户原话的形式化）：
`通讯协议 → 挂钩 → 采集服务 → 绑定 → 站位 → 属于 → 产线 → 属于 → 车间`

**对照 ThingsBoard**：

| 本系统 | ThingsBoard |
|--------|-------------|
| Workshop / Line / Station | Asset 资产层级 |
| DeviceProfile | Device Profile |
| Collector（采集服务） | Gateway Connector 实例 |
| CommChannel（通讯块） | Connector 传输/连接配置 |
| DataPoint（数据块） | Connector 的 attributes/timeseries 映射 |

---

## 二、领域独立化（解耦）

每个域 = 独立表 + 独立 API 命名空间 + 独立 UI 模块；**域间只通过 ID 引用**，互不侵入。

| 域 | 实体 | UI 模块 | 表 |
|----|------|---------|----|
| **组织域** | Factory / Workshop / Line / Station | 组织架构（树） | `factory` `zone` `line` `station` |
| **设备域** | DeviceProfile | 设备类型 | `device_profile` |
| **采集域** | EdgeGateway / Collector / CommChannel / DataPoint | 采集服务 | `edge_gateway` `collector` `comm_channel` `data_point` |
| **字典域** | DefectCode / AlarmCode | 数据字典 | `defect_dict` `alarm_dict` |
| **生产域** | Shift / Target / Threshold | 生产配置 | `shift` `work_order` 等 |

> 解耦带来的好处：换协议只动采集域、调设备类型只动设备域、改组织结构不碰采集逻辑。

## 三、数据结构

完整 TS 定义见 `packages/data-platform/src/config-domain/types.ts`。关键实体：

```ts
Station   { id, lineId, name, seq, deviceType, stdCycleMs }
Collector { id, stationId, gatewayId, name, protocol, pollMs, enabled }
CommChannel { id, collectorId, protocol, config }   // config 随协议而变（可辨识）
DataPoint { id, collectorId, name, canonicalField, address, dataType, scale, mode, funcCode }
```

通讯块 `config` 按协议：
- `modbus_tcp`: `{ host, port, unitId }`
- `opcua`: `{ endpoint, securityPolicy }`
- `mqtt`: `{ brokerUrl, topic }`
- `simulator`: `{}`

数据块 `canonicalField` 取值：`passCount | failCount | cycleTimeMs | equipmentStatus | wipBuffer | metrics.<key>`，即映射到《对外接口与通道设计》的 canonical 遥测字段。

## 四、配置 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/config/snapshot` | 整库快照（首屏一次拉全） |
| GET | `/api/config/:entity` | 列表 |
| GET | `/api/config/:entity/:id` | 单条 |
| POST | `/api/config/:entity` | 新建 |
| PUT | `/api/config/:entity/:id` | 更新 |
| DELETE | `/api/config/:entity/:id` | 删除（**级联**清理下级） |
| GET | `/api/config/gateway/:id/collectors` | 某网关采集配置（edge-gateway 拉取生效） |

`:entity` ∈ `factories/workshops/lines/stations/deviceProfiles/gateways/collectors/channels/dataPoints/defectCodes/alarmCodes/shifts/targets/thresholds`

**级联删除**：删产线→清站位→清采集服务→清通讯块/数据块，保证配置一致。

## 五、存储

| 环境 | 实现 | 说明 |
|------|------|------|
| 开发/默认 | **文件 JSON**（`CONFIG_FILE`，默认 `data/config.json`） | 免数据库，admin 修改即落盘，重启仍在 |
| 生产 | **PostgreSQL** | 表结构见 `docker/postgres/init.sql`，替换 `ConfigStore` 实现即可 |

启动时由种子初始化（与 masterdata 一致），含完整 6 线 34 站 + 采集服务/通讯块/数据块示例。

## 六、闭环：配置即生效

```
admin-web ──CRUD──▶ /api/config/* ──▶ ConfigStore（文件/PG）
                                          │
              data-platform 启动 ──读取──▶ buildWorkshopDef() ──▶ Aggregator 运行时拓扑
              edge-gateway ──GET /gateway/:id/collectors──▶ 采集 Profile（Modbus 寄存器映射）
```

- **data-platform**：启动时 `buildWorkshopDef(configStore, 'W01')` 把配置转成运行时拓扑；
  之后 admin-web 任何改动经 `ConfigStore.onChange` 触发 `Aggregator.applyTopology()` **热重载**——
  新增/删除产线站位即时生效，存活站位**保留已累计计数**，无需重启。
- **edge-gateway**：`GET /api/config/gateway/:id/collectors` 拉取本网关的采集服务 + 通讯块 + 数据块，驱动 Modbus 等真实采集（无需改代码，改配置即可）。

## 七、UI 模块

| 模块 | 交互 |
|------|------|
| **组织架构** | 左侧资产树（车间>产线>站位），右侧详情 + 新建下级/编辑/删除 |
| **采集服务** | 采集服务列表；每行「通讯/数据块」抽屉：通讯块按协议表单 + 数据块点表 CRUD |
| **设备类型** | 设备档案 + 动态 metrics 字段编辑 |
| **数据字典** | 缺陷/停机代码两个 Tab |
| **生产配置** | 班次/目标/阈值三个 Tab |

技术：React 18 + Ant Design 5 + Vite；通用 `CrudTable`（配置+表单驱动）覆盖表型模块，组织树与采集抽屉为定制组件。

## 七·补 RBAC 权限

| 角色 | 权限 |
|------|------|
| **admin 厂区管理员** | 全部 + 用户管理 |
| **ops 运维** | 读写配置（组织/采集/设备/字典/生产），无用户管理 |
| **viewer 只读** | 仅查看，写操作按钮隐藏；后端 403 兜底 |

- 登录 `POST /api/auth/login` → 签发 JWT（HS256，8h）；前端存 localStorage，请求带 `Authorization: Bearer`。
- 服务端 `onRequest` 守卫：读 `/api/config/*` 需 viewer+，写需 ops+，`/api/auth/users*` 需 admin。
- 机器端点 `/api/config/gateway/:id/collectors` **免鉴权**（边缘网关无账号），已内联 deviceType。
- 大屏 `/api/workshop`、WS、MES Webhook 保持公开。
- 演示账号：`admin/admin123` · `ops/ops123` · `viewer/viewer123`。

## 八、运行

```bash
# 后端（含配置 API）
cd packages/data-platform && npm install && npm start   # :8080
# 管理后台
cd packages/admin-web && npm install && npm run dev      # :5174
# 指定后端地址（默认 http://localhost:8080）
VITE_API_BASE=http://localhost:8080 npm run dev
```

Docker：`docker compose up -d` 含 admin-web（nginx 托管），见 `docker/`。
